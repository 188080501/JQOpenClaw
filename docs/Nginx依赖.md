# Nginx 依赖说明

## 1. 整体说明（先看这个）

- 本文档描述 `JQOpenClawNode` 截图文件服务的推荐落地方式。
- 默认假设部署环境是 **Ubuntu**（例如 Ubuntu 22.04/24.04，`apt` 安装 Nginx）。
- 当前方案使用 Nginx 原生能力做最简实现，目标是接入简单、运维成本低。
- 文件服务器是可选项，不是必须项：只有使用 `system.screenshot` 并希望返回可访问 URL 时才需要。

## 2. 部署方式（Ubuntu）

### 2.1 前置准备

1. 确保已安装 Nginx。
2. 准备证书文件（如需 HTTPS）。
3. 在服务器创建数据目录（示例：`/srv/openclaw/data/`）。

示例命令：

```bash
sudo apt update
sudo apt install -y nginx
sudo mkdir -p /srv/openclaw/data
sudo chown -R www-data:www-data /srv/openclaw/data
sudo chmod -R 750 /srv/openclaw/data
```

### 2.2 手动添加 conf（关键）

假设仓库路径为 `<repo_path>`，将本仓库的 `docs/data.conf` 手动加入你的 Nginx 配置。

```bash
sudo cp <repo_path>/docs/data.conf /etc/nginx/conf.d/jqopenclaw-data.conf
sudo nano /etc/nginx/conf.d/jqopenclaw-data.conf
```

需要按实际环境调整以下配置项：

- `ssl_certificate` / `ssl_certificate_key`
- `set $auth_token`
- `alias /srv/openclaw/data/`
- `listen` 端口
- `client_max_body_size`

### 2.3 校验与重载

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### 2.4 Node 侧配置

在 `JQOpenClawNode` 的 `config.json` 中设置：

- `fileServerUrl`：例如 `https://files.example.com:7443`
- `fileServerToken`：与 Nginx 中 `X-Token` 校验值一致

## 3. Nginx 与 JQOpenClaw 的关系

- `JQOpenClawNode` 与 Gateway 的核心协议链路（WebSocket 握手、invoke 调用）不依赖 Nginx。
- Nginx 在当前项目中只承担截图文件服务角色：
  - `PUT /upload/*`：接收上传
  - `GET /files/*`：对外访问
  - `X-Token`：上传鉴权
- 所以它是“可插拔文件服务实现”，不是核心协议组件。

## 4. 为什么走 URL 上传，而不是 base64

当前采用“上传二进制文件并返回 URL”，是综合取舍后的折中结果：

- base64 会增加体积，放进 JSON 后传输更重。
- 大体积 JSON 更容易触发网关/代理限制与超时。
- 控制消息与媒体数据分离后，链路和排障更清晰。
- 可复用现有 Nginx/静态文件服务能力（缓存、限流、访问控制、审计）。

## 5. 关联文件与参数

- 配置模板：`docs/data.conf`
- Node 配置：`fileServerUrl`、`fileServerToken`

URL 规则：

- 上传地址：`<fileServerUrl>/upload/<auto-generated-filename>.jpg`
- 访问地址：`<fileServerUrl>/files/<auto-generated-filename>.jpg`

## 6. HTTP 接口约定

### 6.1 上传接口（鉴权）

- 方法：`PUT`
- 路径：`/upload/<filename>`
- Header：`X-Token: <token>`
- Body：文件二进制

示例：

```powershell
curl -k -X PUT ^
  -H "X-Token: CHANGE_ME_TOKEN" ^
  --data-binary @test.bin ^
  https://127.0.0.1:<custom_port>/upload/test.bin
```

### 6.2 访问接口（默认不鉴权）

- 方法：`GET`
- 路径：`/files/<filename>`

示例：

```powershell
curl -k ^
  https://127.0.0.1:<custom_port>/files/test.bin ^
  -o test.bin
```

## 7. 状态码约定

| 状态码 | 说明 |
| --- | --- |
| 200 | 下载成功 |
| 201 / 204 | 上传成功 |
| 400 | 非法路径（包含 `..`） |
| 401 | 上传未提供 Token |
| 403 | 上传 Token 错误 |
| 404 | 文件不存在 |
| 405 | 请求方法不允许 |

## 8. 风险与建议

- 若 `GET /files/*` 不鉴权，URL 可被直接访问，需评估暴露范围。
- 建议使用高强度随机 token，避免提交到仓库。
- 生产环境建议使用可信证书并启用严格 TLS 校验。
- 如需更强保护，可增加 IP 白名单、签名 URL 或对象存储鉴权。
